# Architecture

Hermes Mobile is an Android port of the [Hermes Desktop](https://github.com/NousResearch/hermes-agent) Electron client. This document describes the architecture as of 2026-06, after the parity-plan pivot described in `~/.claude/plans/groovy-fluttering-island.md`.

## TL;DR

- **Single source of truth for Android code:** [`android-runner/`](../android-runner/) is the canonical home for every Hermes Kotlin file. The live tree at `apps/mobile/android/` is regenerated by [`apps/mobile/scripts/setup-android.sh`](../apps/mobile/scripts/setup-android.sh) on every build. CI's [`script/check-kotlin-drift.sh`](../script/check-kotlin-drift.sh) fails the build on any divergence.
- **UI:** Jetpack Compose. No WebView. No Capacitor. No vendored React renderer.
- **IPC:** A single Kotlin singleton — `HermesApi` — is consumed directly by the Compose UI. There is no `window.hermesAPI` JavaScript bridge.
- **Python backend:** Subprocess (Termux or bundled `python-build-standalone`) running `hermes-agent`, exposed at `127.0.0.1:8642` (or a remote URL over HTTPS / SSH tunnel). HermesApi talks to it via OkHttp + SSE.
- **Distribution:** Sideload + F-Droid + GitHub Releases. No Play Store in v1.

## Why no WebView?

The original plan was a Capacitor+WebView shell that vendored the desktop's React renderer. The intent was "free" desktop parity: re-render the desktop's 25K-LOC React tree, replace the Electron preload with a Capacitor plugin, ship. It would have been the lowest-effort path to a 1:1 desktop clone.

We pivoted. The reasons:

1. **The desktop's renderer isn't Android-native.** Every screen in the React tree assumes a desktop window with a sidebar + content area + status bar. Phones have bottom-nav, 360dp-wide viewports, and back buttons. A 1:1 React port would either look wrong on phones or require a parallel "mobile" React tree to ship — which is most of the work of writing the Compose version anyway.
2. **The IPC was 80% there.** `HermesApi.kt` already had 44 native methods covering install, gateway, OAuth, SSH, voice, memory, chat, sessions, and 16 SharedFlows. The remaining ~110 methods are mostly 5-10 line file-IO wrappers — not a rewrite.
3. **A typed Kotlin API is better than a JS bridge.** `hermesAPI.chat.sendMessage(...)` becomes `hermesApi.sendMessage(...)` with proper coroutine semantics, structured concurrency, and no `Promise` shim. Screens that need a stream subscribe to a Kotlin `SharedFlow` and the type checker enforces the event shape.
4. **Maintenance.** A Capacitor bridge is a translation layer between two runtimes. The Compose path is a single runtime. The plan file's "Why this plan will hit parity in 16 weeks" section quantifies the win.

### Why `apps/mobile/android/app/build.gradle` is in the tree, not regenerated

Capacitor's `npx cap add android` generates the entire `apps/mobile/android/` tree from the `package.json` template. The build.gradle is part of that scaffolding. With Capacitor gone, the tree is no longer regenerated; the live `build.gradle` is hand-edited and committed.

`android-runner/app/build.gradle.template` is the source of truth for new gradle additions. Changes should be made to both files (or the live one, with the template updated to match when a fresh `cap add android` is ever needed again — which it won't be).

## HermesApi surface

See `~/.claude/plans/groovy-fluttering-island.md` Phase 1 for the full list of methods. The high-level shape:

```kotlin
class HermesApi(context: Context) {
    // Platform info
    fun getPlatform(): PlatformInfo
    fun getAppVersion(): String
    fun quitApp()

    // Connection
    fun getConnectionConfig(): ConnectionConfig
    fun setConnectionConfig(mode: String, remoteUrl: String, apiKey: String)
    fun startSshTunnel(): Boolean
    fun stopSshTunnel()
    fun isSshTunnelActive(): Boolean

    // Install
    fun checkInstall(): HermesInstaller.CheckResult
    fun startInstall()
    fun verifyInstall(): Boolean
    fun getHermesVersion(): String?
    fun runHermesDoctor(): String
    suspend fun runHermesUpdate(): HermesInstaller.UpdateResult

    // Gateway
    fun startGateway()
    fun stopGateway()
    fun gatewayStatus(): Boolean

    // Chat (suspend; streaming)
    suspend fun sendMessage(...)
    fun abortChat()
    val chatChunk: SharedFlow<ChatChunk>
    val chatReasoningChunk: SharedFlow<...>
    val chatToolProgress: SharedFlow<...>
    val chatUsage: SharedFlow<...>
    val chatDone: SharedFlow<...>
    val chatError: SharedFlow<...>

    // Memory / Soul / Sessions / ... (per Phase 1)

    // Mobile-only
    fun requestIgnoreBatteryOptimizations(): Boolean
    fun getBatteryOptStatus(): BatteryOptStatus
    fun setStartOnBoot(enabled: Boolean)
    fun getStartOnBoot(): Boolean
    fun setCrashReportingEnabled(enabled: Boolean)
    fun getCrashReportingEnabled(): Boolean
    fun startVoiceCapture(): VoiceCaptureStart?
```

## Adding a new Hermes method or screen

1. Edit the source file in `android-runner/app/src/main/kotlin/com/nousresearch/hermes/`.
2. Run `bash apps/mobile/scripts/setup-android.sh` to sync.
3. Verify with `bash script/check-kotlin-drift.sh`.
4. `./gradlew :app:assembleDebug` to build.
5. Commit both `android-runner/` and `apps/mobile/android/` changes.

## Build & test

```bash
# One-time setup
pnpm install
bash apps/mobile/scripts/setup-android.sh
bash script/check-kotlin-drift.sh

# Build
cd apps/mobile/android
./gradlew :app:assembleDebug          # debug APK
./gradlew :app:assembleRelease        # release APK (needs keystore)
./gradlew test                        # unit tests
./gradlew connectedAndroidTest        # instrumented tests (needs emulator)
```

## Distribution

| Channel | Build | Update mechanism |
|---|---|---|
| **Sideload (GitHub Releases)** | `assembleRelease` (signed with `keystore/release.jks`) | In-app: Settings → Check for Updates reads `latest.json` from the release |
| **F-Droid** | `f-droid/build.sh` (unsigned; F-Droid signs) | Auto-update on F-Droid client |
| **Manual APK** | `assembleDebug` | None; user reinstalls |

See `keystore/README.md` for the release key fingerprint and `f-droid/metadata/com.nousresearch.hermes.yml` for the F-Droid entry.
